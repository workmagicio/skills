## Interpretation playbook — canonical language for common patterns

<callout emoji="🎯">
**First pick the framework by scenario type (Maximize vs Target) — this determines how you explain it.**
- **Maximize (Outcome Max)**: the core isn't any single saturation curve, but **relative marginal-efficiency reallocation across channels** — MBO compares each channel's "return on the next dollar" and moves budget from channels with low marginal returns to those with high ones, until each channel's marginal return is roughly **equalized**. The interpretation should speak to the relative gaps **between** channels, not just describe one channel as "steep/flat" on its own.
- **Target Achievement**: closer to single-channel curve position — hitting a given goal (e.g., 3.5x ROAS) with the least spend, where each channel's curve position directly drives the increase/decrease, so the single-channel language below applies more.
The canonical language below is written from a single-channel perspective by default; **for Maximize scenarios, lead with relative / cross-channel phrasing** (see the Maximize notes in the "increasing/cutting channel X" rows).
</callout>

| **Pattern** | **Business-language explanation** |
|-|-|
| MBO recommends increasing channel X | **Maximize:** "Under your current allocation, {X}'s **next-dollar return is higher than the other channels'** — so MBO shifts budget toward it, until the marginal returns across channels equalize. It's not that {X} itself is 'performing well', but that relative to the other channels, a dollar spent on {X} is more worthwhile right now." **Target:** "{X}'s saturation curve is still steep at the current spend level, so adding budget still yields meaningful incremental return." |
| MBO recommends cutting channel X | **Maximize:** "Cutting {X} **doesn't mean it's bad** — its ROAS is very likely still positive. The reason is **opportunity cost**: under the current allocation, a dollar spent on another channel has a **higher marginal return**, so MBO shifts part of {X}'s budget away to raise total output. Make clear it's 'worth more elsewhere right now relative to other channels', not '{X} is saturated / underperforming'." **Target:** "{X} is near saturation; its curve is already flat at the current spend level, so each additional dollar yields little incremental return, hence the reduction." |
| High average ROAS but low marginal ROAS | "Average ROAS blends efficient early spend with saturated late spend into one number. Channel X's average ROAS looks healthy, but at its current spend level the marginal ROAS — return on the next dollar — has collapsed. That's why MBO is suggesting a cut." |
| Baseline (organic) sales are large vs paid media sales | "A big share of your sales are organic / recurring demand that exists independently of paid media. MBO doesn't optimize that baseline — it only reallocates the paid portion. Reading the recommendation requires looking at the paid component separately." |
| **Total projected sales is LOWER than reference, even though the goal is "maximize sales" (or similar positive direction)** | **This is almost always a baseline issue, not a recommendation issue.** Total sales = baseline (organic) sales + paid media sales. Decompose before explaining:  <br/>• **Paid media component**: is it up or down vs reference? If up (and ROAS / marginal ROAS improved) → **the reallocation IS working as intended**; this is the part MBO optimizes.  <br/>• **Baseline component**: if down → that's where the total drop comes from. MBO doesn't optimize baseline — it's projected from organic trend + seasonality.  <br/>• **Net**: surface the decomposition explicitly. Say something like "Paid media sales are up \$X (+Y%) and ROAS improved from A → B — the reallocation is doing its job. Total is down \$Z because the model projects baseline (organic) demand to be Q% lower next period (seasonal / organic decline). MBO can't optimize baseline, so this drop is outside the recommendation's control."  <br/>• **If low backtesting confidence flag is present**, additionally note: "Treat the absolute projection as directional rather than literal."  <br/>**Never say** "the reference baseline reflects channel mix" or "the optimizer finds a more efficient distribution" without first checking baseline — that explanation is generic and often misleads. |
| **User has manually locked several tactics; total projected sales is lower than reference** | **Don't blame the user's own locks for the drop.** The locks are the user's intent, not a limitation to explain away. Saying "because you locked X, Y, Z, this isn't the best plan overall" is condescending and unhelpful — the user already knows what they locked.  <br/>Correct framing:  <br/>• Treat locks as fixed boundary conditions; don't list them as the cause  <br/>• Run the baseline / paid media decomposition (see row above) — the drop is usually baseline, not the locks  <br/>• Within the unlocked tactics, explain what MBO did do (which got more, which got less, why)  <br/>• Only mention the locks if the user explicitly asks "why didn't the model optimize tactic X" — and even then, name them neutrally as scope, not as a problem  <br/>**Never say**: "the model is solving best plan given these locks, not best plan overall, that's why total sales drop" — this is the anti-pattern Iris flagged. |
| Recommendations differ from a run a few days ago (same inputs) | "Saturation curves refresh daily. Even a few days can shift a channel's curve, particularly near the saturation inflection. New lift test results also recalibrate the affected channels. Expected behavior." |
| MBO vs attribution dashboard for same period | "MBO shows model-estimated outcomes at the reference spend; dashboard shows realized actuals using last-click / iDDA. Two different methodologies, expected to differ. Dashboard for performance tracking, MBO for allocation." |
| iDDA strong recently, MBO forecast lower | "iDDA shows realized iROAS — fluctuates with promos, creative, seasonality. MBO fits a stable saturation curve to smooth those spikes, predicting the sustainable future. The gap is logical, not a discrepancy." |
| Backtesting accuracy < 70% | "Backtesting accuracy is [X]%, below recommended 70%. Use these recommendations as directional guidance. Your CS team can review what's driving the lower accuracy." |
| Lift test iROAS vs MBO marginal ROAS differ | "Lift test measures incremental ROAS (causal contribution at the tested spend). MBO marginal ROAS measures return on the next dollar at the recommended spend. They're complementary measurements, not competing ones — lift test is actually used to calibrate MBO." |
| Reference period spend = 0 for a channel | "Channel X had zero spend in the reference window — MBO can't model what it would do at zero baseline, so it's excluded from the recommended allocation. To include it, run a scenario with a reference window where you actually spent on it." |
| Insufficient data for saturation curve | "Channel Y doesn't have enough historical data to build a reliable saturation curve, so MBO held its budget at the baseline (no change). Once more history accumulates, future scenarios can optimize it." |
